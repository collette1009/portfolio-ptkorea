VOC 전수 분류를 통한 CX 구조 문제 발굴 및 개선 실행

같은 CS 문의가 반복될 때 건별 처리가 아닌 구조적 원인을 찾는 것이 출발점이었습니다. 두 케이스, 두 가지 결론 — 프로모션 오인 VOC 173건 분류로 82% 동일 원인을 규명했고, 배송 제약 상품이 "품절"로 잘못 표시되는 구조 문제를 발견해 CTO 직접 제안·당일 구현으로 CS 문의 일 6~10건→0건을 달성했습니다. 이 해법은 상품운영파트가 프로모션 안내에도 요청 없이 동일하게 확산 적용하며 단일 제안이 사내 UI 개선 기준이 됐습니다.
#VOC데이터분석 #CX개선실행 #구조적원인규명 #개발팀협업

CS 문의는 처리되었지만, 반복은 멈추지 않았습니다

문제 1 — 프로모션 오인 VOC가 반복 유입
프로모션 조건을 오인한 고객 문의가 지속적으로 유입됐습니다. 건별로 처리하고 있었지만 동일한 유형이 계속 반복됐고, 구조적 원인을 분석한 적이 없었습니다.
문제 2 — 배송 제약 상품이 "품절"로 잘못 표시
배송 가능일이 다른 상품이 장바구니에 함께 담기면 결합 제약 상품이 "품절"로 잘못 표시됐습니다. 고객은 이유를 모른 채 장바구니를 이탈했고, CS로는 일 6~10건의 문의가 반복 유입됐습니다.
공통 문제 — CS가 처리하고 끝나는 구조
두 케이스 모두 문의를 처리하고 끝내는 구조였습니다. 같은 문의가 반복되는 이유를 데이터로 분석하고, 원인을 제거하는 CX 개선으로 연결하는 구조가 없었습니다.

건별 처리 대신 데이터로 구조를 바꿨습니다

1. VOC 전수 분류 — 프로모션 오인 173건
82% 동일 구조적 원인 규명 → 개발 협의 → 운영 기준 직접 수립
프로모션 오인 VOC 173건을 유형별로 전수 분류했습니다. 분석 결과 82%가 동일한 구조적 원인에서 발생하고 있었습니다 — 프로모션 조건 미충족 시 안내 문구가 표시되지 않는 구조였습니다. 이를 데이터로 정리해 이커머스 리드에게 개선안을 제안·협의했습니다. 다만 당시 인입 감소 추세를 감안해 즉시 개발 대신 운영으로 대응하는 방향으로 결론이 났고, 초입 고객에게만 선제 AS를 발송하는 운영 기준을 직접 수립해 동일 유형 반복 문의를 줄였습니다. 데이터로 원인을 증명하는 것이 팀의 판단 기준을 만들었습니다.
2. 배송 VOC 패턴 분석 → CX 구조 재정의
화요일 배송 상품 "품절" 오표시 → CTO MVP 제안 → 당일 구현 → CS 문의 0건
화요일에만 배송되는 상품(연두부 등)이 다른 상품과 함께 장바구니에 담기면 해당 상품이 "품절"로 잘못 표시되는 구조적 문제를 VOC 패턴에서 발견했습니다. 고객은 실제로 품절이 아닌데 품절로 오인해 CS로 문의하거나 이탈했고, 일 6~10건의 반복 문의로 이어졌습니다. 이 문제를 데이터 기반으로 재정의하고 CTO에게 직접 MVP를 제안했습니다 — 장바구니 내 해당 상품 옆에 "⚠️ 화요일 도착 / 배송 가능일이 제한된 상품이라 나누어 주문이 필요할 수 있어요." subtext를 표시하는 방식. CTO가 즉시 수용하고 SW개발팀이 당일 구현, 유사 상품(연두부 85g 등)으로도 확대 적용됐습니다. 관련 CS 문의는 일 6~10건→0건으로 완전 해소됐습니다.
3. 개선 사이클 정착
VOC 리뷰 → 원인 규명 → 개선 실행 → 효과 측정
두 케이스를 통해 VOC 리뷰 → 구조적 원인 규명 → 유관부서 협의 → 개선 실행 → 효과 측정의 CX 개선 사이클을 정착시켰습니다. 반복되는 CS 문의를 "처리"가 아닌 "차단"하는 구조로 전환했습니다.

근거

1. 실제 적용 화면 — 주의상품 subtext + 프로모션 subtext 동시 확인
화요일에만 배송되는 두부 상품에 "⚠️ 화요일에만 도착 가능한 상품이에요. 나누어 주문이 필요할 수 있어요." subtext가 적용된 실제 화면입니다. 아래 짜먹는 멀티비타민에는 "2개 담으면 그 중 1개는 무료"라는 프로모션 subtext가 함께 보입니다 — 상품운영파트가 주의상품 패턴을 보고 프로모션 안내에도 동일하게 자체 적용한 결과입니다. 단일 제안이 사내 UI 기준으로 확산된 것을 한 화면에서 확인할 수 있습니다.
2. 기술 제약 기반 예외 처리 재정의 및 MVP 설계
모든 실패 조건을 세분화하는 방식 대신, API 응답값으로 실제 판단 가능한 조건만 남겨 구현 가능한 범위 안에서 예외 처리 구조를 재설계했습니다. 기술 제약을 전제로 한 실행 가능한 최소 단위(MVP) 해법으로 개발팀과 합의했습니다.

배운 점

1
VOC는 처리 대상이 아니라 팀의 판단 기준을 바꾸는 데이터입니다. 173건 분류로 82%가 같은 원인이라는 것을 증명했을 때, 개발을 진행하지 않더라도 "어떤 고객에게 어떤 AS를 먼저 보낼지"를 결정하는 기준이 생겼습니다.
2
개발이 안 된다고 해도 운영 기준을 만들 수 있습니다. 173건 케이스에서 즉시 개발이 불필요하다는 결론이 났을 때, 데이터가 보여준 원인(초입 고객의 조건 인지 부족)을 기반으로 초입 고객 선제 AS 발송 기준을 직접 수립했습니다. 제안이 받아들여지지 않아도 실행은 이어집니다.
3
CX 개선은 데이터로 근거를 만들어야 빠르게 실행됩니다. 배송 상품 케이스는 "일 6~10건 반복 문의, 특정 상품 조합 패턴"을 데이터로 정리해 CTO에게 제안했을 때 당일 구현이 됐습니다. 같은 문제를 정성적으로만 전달했다면 결과가 달랐을 것입니다.
4
좋은 구조는 요청 없이도 다른 곳에 퍼집니다. 주의상품 subtext 방식을 상품운영파트가 스스로 프로모션 안내에 적용했습니다. 강요 없이 다른 팀이 따라 한다는 건 그 솔루션이 진짜 문제를 해결했다는 신호입니다.
