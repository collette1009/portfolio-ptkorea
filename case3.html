고객 VOC에 숨어 있던 의도를 필수 선택값으로 구조화해, AS 재확인 왕복을 95% 이상 없앤 사례입니다. 데이터 품질은 처리 단계가 아닌 수집 단계에서 결정된다는 관점에서 접근했고, 중복 DB 필드를 활용하는 스펙으로 개발 범위를 최소화하면서 빠르게 구현했습니다.

반복 확인으로 고객과 운영 모두 지치고 있었습니다

문제 1 — 고객 의도가 자유 텍스트에 묻혀 있었음
고객이 환불·교환 의도를 자유 텍스트로만 남겨 추가 확인이 필요한 구조였습니다. CS 담당자가 매건 고객에게 재확인해야 했고, 고객은 1:1 문의에 다시 답변해야 했습니다.
문제 2 — 재확인 왕복으로 리드타임 지연·고객 피로 증가
반복 확인으로 고객 피로도와 운영 리소스가 함께 증가했습니다. CRM에는 답변 작성일과 사실상 중복되는 확인일시 필드가 기록되어, 의미 없는 데이터만 누적되는 구조였습니다.
문제 3 — CS 처리 속도가 사람의 판단 속도에 좌우됨
자유 텍스트에 의존하면 처리 속도와 정확도가 담당자 개인의 판단에 좌우됩니다. 담당자마다 해석이 달라지는 품질 편차 문제도 있었습니다.

데이터 수집 단계부터 구조를 바꿨습니다

1. 데이터 품질의 출발점 재정의
처리가 아닌 수집 단계에서 품질 결정
데이터 품질은 처리 단계가 아니라 수집 단계에서 결정된다고 보고, 처리 중 추가 질문을 늘리기보다 입력 구조 자체를 바꾸는 방향으로 접근했습니다. "질문을 줄이는 설계"가 운영 비용을 줄인다는 원칙을 적용했습니다.
2. 중복 필드 제거 및 의미 있는 데이터로 교체
확인일시 → 요청유형 선택값으로 전환
답변 작성일과 사실상 중복되던 확인일시 필드를 제거하고, 해당 위치에 고객의 요청 유형(환불/교환) 선택값을 배치했습니다. 불필요한 DB 필드를 제거하고 의미 있는 데이터 구조로 전환한 것입니다. 개발 범위를 최소화하는 스펙을 직접 정의해 구현 속도를 높였습니다.
3. 입력 구조 재설계 및 개발팀 협의
스펙 직접 정의 → SW개발팀·IT팀 협의 → 구현
고객 문의 등록 단계에서 환불·교환 의도를 필수 선택값으로 수집하고, CRM에 구조화된 데이터로 저장되도록 설계했습니다. 중복 필드 자리를 활용하는 방식으로 개발 범위를 최소화하는 스펙을 직접 정의하고, SW개발팀·IT팀과 협의해 DB 컬럼 추가 및 UI 구현을 빠르게 완료했습니다.

3단계의 개선 과정

1. 기존 입력 구조 — 문제 현황
기존 CRM의 환불·교환 입력 화면 — 자유 텍스트 의존 구조와 중복 필드(확인일시)가 있던 상태입니다. 고객 의도를 파악하기 위해 매건 재확인이 필요했습니다.
2. 입력 구조 재설계
답변 작성일과 중복되던 확인일시 필드를 제거하고, 해당 위치에 고객 의도 데이터를 매핑했습니다. 개발 범위를 최소화하는 스펙을 직접 설계해 빠르게 반영했습니다.
3. 요청 유형 선택값 반영
고객이 문의 등록 단계에서 환불·교환 의도를 직접 선택하도록 바꾸고, 선택된 값이 후속 처리 기준으로 바로 활용되도록 설계했습니다.

배운 점

1
질문을 줄이는 설계가 운영 비용을 줄였습니다. 처리 단계에서 확인을 추가하는 것보다 수집 단계를 구조화하는 것이 훨씬 효과적이었습니다. CX 개선의 출발점은 처리 속도가 아닌 데이터 수집 품질이었습니다.
2
고객 의도는 해석이 아닌 선택으로 수집해야 합니다. 자유 텍스트에 의존하면 처리 속도와 정확도가 사람의 판단에 좌우됩니다. 필수 선택값으로 구조화했을 때 처리 흐름이 끊기지 않고 자연스럽게 이어졌습니다.
3
최적화는 기능 추가보다 구조 점검에서 시작됩니다. 중복 필드를 발견하고 그 자리를 활용한 이 접근법이 개발 부담을 최소화하면서도 큰 효과를 냈습니다. 시스템적 사고가 CX 개선을 가능하게 했습니다.
