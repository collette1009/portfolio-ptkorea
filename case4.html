장바구니 주문 오류 Fail-soft UX 설계 — 이유를 모른 채 이탈하는 고객을 흐름으로 이어가기

배송 가능일이 다른 상품이 장바구니에 함께 담기면 주문이 차단되는 상황에서, 고객이 이유를 모른 채 이탈하는 CX 문제를 발견했습니다. 기술적 제약으로 모든 예외를 처리하기 어려운 상황에서, 구현 가능성을 전제로 설계 방향을 전환해 Fail-soft UX를 직접 기획하고 개발팀과 합의했습니다. 관련 CS 문의 일 2~3건 → 0건.
#FailSoftUX #고객여정분석 #MVP기획 #ExceptionDesign

고객은 이유를 몰랐고, 내부에도 기준이 없었습니다

문제 1 — 주문 차단 이유가 고객에게 전혀 전달되지 않았음
배송 가능일이 다른 상품이 함께 담기면 주문이 차단됩니다. 그런데 고객에게는 이유가 전달되지 않아 시스템 오류처럼 인식됐습니다. 주문에 실패한 고객은 재시도 없이 이탈했고, CS 문의로 유입됐습니다.
문제 2 — 무료배송 조건 충족을 위해 상품을 더 담다가 이 상황에 걸리는 패턴
무료배송 기준(4만원)을 맞추려 여러 상품을 함께 담는 패턴에서 이 오류가 자주 발생했습니다. 장바구니 이탈과 잠재 매출 손실로 직결되는 CX 문제였습니다. 내부에서도 "상품을 하나씩 빼보세요"라는 안내가 전부였습니다.
문제 3 — 외부 API 제약으로 이상적 예외 처리 로직 구현이 불가한 상태
외부 시스템(API) 응답 제약으로 인해, 기획 단계에서 의도한 수준의 세부 예외 처리 로직을 그대로 구현하기 어려운 기술적 한계가 있었습니다. 접근 방법 자체를 바꿔야 했습니다.

제약 안에서 실행 가능한 최선안을 찾았습니다

1. 구현 가능성을 포함한 재설계
이상적 로직의 한계를 먼저 확인하고 방향을 바꿨습니다
초기에는 모든 실패 조건을 시스템적으로 구분하려 했지만, 구현 가능성과 일정 측면의 현실적 한계를 확인했습니다. 구현 가능성까지 포함해 예외 처리 방식 자체를 다시 설계해야 한다고 판단하고, 기술 제약을 전제로 한 MVP 방향으로 전환했습니다.
2. API 응답값 기반 MVP 설계
기술 제약을 전제로 한 실행 가능한 최소 단위 설계
API 응답값으로 판단 가능한 조건만을 기준으로 실패 유형을 재정의했습니다. 기술 제약을 전제로 한 실행 가능한 최소 단위(MVP) 해법으로 방향을 전환해 개발팀과 합의했습니다. 완벽한 처리보다 고객이 상황을 이해하고 흐름을 이어갈 수 있는 것이 우선이라는 원칙을 적용했습니다.
3. Fail-soft UX 설계 및 운영 기준 정비
오류 상황에서도 고객이 흐름을 이어가도록 설계
주문 실패 시 세부 원인을 모두 설명하지 않더라도, 왜 지금 진행할 수 없는지를 이해할 수 있도록 Fail-soft UX를 설계했습니다. 오류 상황에서 FAQ 및 추가 안내로 자연스럽게 연결되도록 흐름을 보완하고, 운영 대응 기준도 함께 정리해 CS 혼선을 줄였습니다.

고객이 이유를 알게 되면서 문의가 사라졌습니다

배송 가능일이 다른 상품이 포함된 경우, 주문 차단 사유를 별도 페이지 이동 없이 장바구니 내에서 즉시 안내하도록 설계
주문 불가 상황에서 고객이 이유를 이해하고 흐름을 이어갈 수 있도록 설계 → 즉각 이탈 방지 및 잠재 매출 손실 구조 해소
오류 인식으로 인한 고객 문의 유입 구조 해소, 기술 제약을 고려한 실행 가능한 개선안으로 개발·운영 간 합의
유사한 제약 상황이 발생하더라도 재사용 가능한 대응 구조 확보 — Fail-soft 접근법의 내부 기준 정립

설계 근거

Fail-soft UX 설계 흐름 — API 제약 기반 MVP 구조
기술 제약(외부 API 응답 한계)을 전제로 처리 가능한 예외 유형을 재정의하고, 각 상황에서 고객에게 전달할 메시지와 다음 흐름을 설계한 문서입니다. 완벽한 로직 대신 고객이 상황을 이해하고 이어갈 수 있는 최소 단위 UX에 집중했습니다.

배운 점

1
CX 관점에서 '오류'는 기술 문제가 아닌 커뮤니케이션 문제입니다. 주문이 차단되는 것 자체보다 고객이 이유를 모른다는 것이 더 큰 문제였습니다. 완벽한 처리보다 이해 가능한 안내가 먼저입니다.
2
기술 제약은 설계 조건이지 포기의 이유가 아닙니다. 이상적 로직을 고수하는 대신, 제약 안에서 실행 가능한 MVP를 정의하는 것이 실제 고객 경험을 개선하는 더 빠른 방법이었습니다.
3
Fail-soft 설계는 운영 기준도 함께 만들어야 완성됩니다. UX 설계 단계에서 CS 대응 기준까지 함께 정리해 배포했을 때, 운영 혼선 없이 일관된 고객 경험이 유지됐습니다.
